# 战城大师当前游戏设计文档

更新日期：2026-07-14

## 1. 文档用途

本文档整理当前 Godot 原型和配置表中已经存在的游戏设计，便于后续修改规则、数值、卡牌、地块、经济和界面流程。

当前工程同时存在两层设计：

- 当前实装原型：主要在 `scripts/app/main.gd` 中运行，包含大厅、编组页、战斗、地块购买、单位推进、胜负结算等。
- 配置表目标规则：主要在 `config/tables/` 和 `runtime/config/` 中，包含单位、技能、关卡、地块类型、价格池、翻开池、卡池、掉落和经济。

后续修改时建议先确定规则属于哪一层。如果希望规则长期可维护，应优先改配置表，并逐步把 `main.gd` 中的硬编码规则迁移到配置读取。

## 2. 游戏定位

`战城大师` 当前是一个移动竖屏、轻策略、卡牌编组加地块争夺的 Godot 原型。

玩家在局外选择出战卡牌，进入局内后通过购买相邻地块、翻开建筑或资源、生成单位和防御建筑来推进领土。动物出生或当前推进目标失效时，锁定最近的敌方建筑作为寻路目标；推进途中只攻击自身攻击范围内的敌对动物或建筑，不为路过目标改道，也不持续重算寻路。锁定建筑死亡或消失后才重新选择最近敌方建筑。普通地块不作为攻击目标；摧毁对方基地或在倒计时结束时靠真实解锁领土和建筑分数取胜。

设计关键词：

- 竖屏移动端
- 大厅加底部导航
- 卡牌编组
- 六边形地块争夺
- 自动单位推进
- 建筑生产和防御
- 价格池和卡池驱动的随机奖励
- 后续可扩展局外养成、抽卡、商店和关卡奖励

## 4. 界面与输入规则

### 4.1 画面基准

| 项 | 当前值 | 位置 |
| --- | --- | --- |
| 设计分辨率 | 720x1280 | `scripts/app/main.gd` |
| Godot 窗口 | 1080x1920 | `project.godot` |
| 拉伸模式 | canvas_items / expand | `project.godot` |
| 主场景 | `scenes/main.tscn` | `project.godot` |
| 主节点 | `Node2D` | `scenes/main.tscn` |

### 4.2 底部导航

| 位置 | 按钮 | 当前功能 |
| --- | --- | --- |
| 左 1 | 商店 | 锁定，提示暂未开放 |
| 左 2 | 编组 | 进入编组页 |
| 中间 | 战斗 | 回到大厅战斗入口 |
| 右 2 | 抽卡 | 锁定，提示暂未开放 |
| 右 1 | 房间 | 进入双边房间页 |

大厅主操作区同时保留 `单人对战` 和 `多人对战`：点击多人按钮立即开始一局 6 人自由混战，不跳转房间页；按钮右上角固定显示小型高对比度 `HOT!` 角标。`多人对战` 主文案必须以整个按钮的几何中心为准进行水平和垂直居中，不能继续为已移除的第二行说明预留偏移；`HOT!` 角标不参与主文案的居中计算。大厅不直接展示玩法规则或奖励说明，详细规则由对应说明页承载。角标属于多人按钮的视觉层和点击范围，不能遮挡按钮主文案。需要组队或自定义人数时，玩家仍从底栏 `房间` 进入双边房间页。

点击大厅基地打开 `设置与账号` 弹窗。弹窗采用独立标题栏、账号区和通用设置区的纵向结构：关闭按钮固定在标题栏右侧；账号输入、登录/注册、连接状态和玩家协议依次排列；音乐与音效位于单独的声音设置区。任何标题、按钮、输入框和状态文字的矩形都不得重叠，玩家协议页复用同一标题栏并保留独立返回按钮。

### 4.3 编组页输入

- 点击页签 `1-5` 切换编组。
- 点击上方出战位或下方所有卡牌列表中的卡牌，都只切换中间详情预览，并播放一次详情区 UI pop 动效。
- 下方未上阵卡牌的详情区显示 `上阵` 按钮；点击后进入替换选择状态，上方所有已上阵卡牌播放 UI 呼吸动效。
- 替换选择状态下，玩家再点击上方某个已上阵卡牌，才会把待上阵卡牌放入该槽位；完成后停止呼吸动效。
- 已上阵卡牌只显示 `升级` 操作，不显示 `上阵` 操作。
- 鼠标滚轮或拖动可滚动所有卡牌列表。

### 4.4 多人战斗输入与房间页

- 底栏最后一个页签进入房间页；房主选择 `1V1 / 2V2 / 3V3`，本地玩家固定在我方 1 号槽位，其余有效槽位可等待邀请玩家或由电脑补位。
- 房间码支持复制，并提供真实玩家接入槽位的接口；在好友、账号和公网房间服务接入前，复制成功不表示远端玩家已经连接。
- 关闭电脑补位时，仍有空槽位则禁止开始；开启后，所有未连接槽位在开局时交给电脑控制。
- 以下地图拖动、点击查看和自动寻敌规则同时适用于大厅 6 人自由混战与双边房间战斗；同边盟友规则只适用于房间模式。
- 多人棋盘继续使用当前战斗页的地块显示尺寸和战斗视窗，不自动缩小整张地图；鼠标左键拖动或单指拖动平移地图。
- 拖动只修改摄像机的视图偏移；单位、建筑、弹道、攻击目标和占领判定始终保留独立的世界坐标，不能因拖动而改变实际位置或战斗距离。
- 按下后移动超过 `12` 设计像素才进入拖动，未超过阈值的按下/抬起按一次点击处理，避免拖动地图时误触地块查看或购买。
- 点击己方可购买地块仍执行购买；点击其他地块或建筑只切换信息查看，不再下达手动攻击指令。
- 动物出生或当前推进建筑死亡、消失、转为盟友时，选择最近的敌方建筑并锁定为推进目标；寻路过程中不因出现新的动物或更近建筑而改道。动物只攻击自身攻击范围内的敌对动物或建筑，玩家拖动地图和查看地块不能改变其推进目标。
- 带有 `flying` 标签的动物可直线飞越地图空白区域；其余地面动物只能沿实际存在且相邻的六边格移动，并使用简单六边格寻路绕过缺失地块。
- 阵营不能只靠颜色识别：单位和建筑生命条使用阵营色，同时显示 `1-6` 槽位编号；本地玩家固定为 1 号，同边槽位互为盟友。

## 5. 编组与卡牌规则

### 5.1 当前实装规则

| 项 | 当前值 |
| --- | --- |
| 编组数量 | 5 |
| 每套编组出战位 | 8 |
| 当前版本卡牌开放 | 初始拥有 8 张普通动物卡、紫色金矿卡、绿色防御塔卡；其他防御塔卡可通过抽卡获得 |
| 卡牌来源 | `config/tables/cards.csv`，包含动物卡、金矿卡和防御塔卡 |
| 卡牌名称来源 | `localization_zh` 文本表 |
| 默认初始化 | 优先放入金矿卡和绿色防御塔卡，再用已拥有动物卡补满编组 |
| 重复限制 | 同一编组内点击已有卡牌时执行交换，避免重复占位 |
| 必带规则 | 每套编组必须包含金矿卡和至少 1 张绿色（common）防御塔卡；尝试替换掉最后一张绿色防御塔时阻止替换，并提示“绿色防御塔必须在卡组中，否则会变成空地” |

配置备注规范：`config/tables/cards.csv` 的 `design_notes` 必须贴近游戏内可见名称，统一使用 `卡牌名：定位/用途/注意事项` 格式，例如 `蜗牛：无技能纯肉盾，属性效率最高。`。

### 5.2 当前卡牌战力公式

当前编组页展示的战力来自原型脚本：

```text
card_power = max_hp + base_damage * 12 + shop_cost * 18 + rarity_rank * 80
deck_power = 所有出战卡牌 card_power 求和
```

品质等级：

| 品质 | rarity_rank | 显示等级 |
| --- | ---: | ---: |
| common | 0 | 1 |
| rare | 1 | 2 |
| epic | 2 | 3 |
| legendary | 3 | 4 |

### 5.3 当前卡牌显示规则

- 动物卡牌的品质、攻击、生命、速度、射程、召唤间隔和技能说明均以 `config/tables/cards.csv` 为权威来源。
- `skill_text` 有内容时，卡牌详情和战斗地块详情原样显示 `skill_text`，并以它作为当前动物技能的优先策划意图。
- `skill_text` 为空但 `skill_trigger / skill_effect / skill_power / skill_cooldown_sec` 有配置时，界面生成一条结构化兜底技能说明。
- 动物卡牌不再把移动速度作为独立属性显示，速度只在战斗计算中继续读取 `move_speed`；如果没有显式 `skill_text`，高速动物才自动生成 `速度快 / 速度超快` 兜底标签。
- 动物需要一次召唤多个时，优先在 `skill_text` 中写 `数量+1` 或 `额外召唤一个...`，战斗生产会按该文本额外生成单位。

### 5.4 当前卡牌美术占位

当前编组卡牌暂用三种已有单位图：

| 品质 | 使用图 |
| --- | --- |
| common | 兔兵图 |
| rare | 狼兵图 |
| epic / legendary | 熊兵图 |

后续需要为每个单位族群补独立头像、卡面或程序化变体。

### 多人出生位置与动物数值反馈

- 多人房间进入战斗前由服务器对全部启用槽位执行一次洗牌；真人与 AI 都参与，房主身份和战斗权威不随出生位置改变。每个客户端从服务器下发的 `local_team_id` 获取本局位置，不能在客户端自行随机。
- 金币与属性反馈绑定动物 `unit_id`，绘制时实时读取动物当前位置，不保存为脱离动物的固定屏幕坐标；动物死亡或已移除时才使用触发瞬间坐标兜底。
- 攻击、生命上限、治疗、护盾、速度、减速和眩晕均使用“对应图标 + 有符号数值/时长”的短时上飘反馈；同一动物短时间内的同类反馈合并，避免遮挡战斗。
- 所有反馈只作用于实际受到效果的动物；群体技能为每个受影响动物分别生成反馈，不能显示在施法者或地块中心。
| 玩家初始金币 | 60 | `gold` |
| 敌方初始金币 | 60 | `enemy_gold` |

### 6.2 当前开局地块布局

经典单人战斗复用房间地图系统的 5 张 `1V1` 对称地图：高原、菱形、沙漏、十字、波纹。每次点击单人对战时在这 5 张地图中等概率随机 1 张；测试和回放可以显式传入 `map_id` 与对局种子。玩家与 AI 各有 36 个初始预归属格；每局先从 3 套合法出生模板中随机 1 套，再把同一模板严格镜像给对手。基地、起始金矿和起始营地始终使用同一套对称变换，不能为单个玩家独立抽取而破坏公平。

预归属只用于战线识别，不等同于已解锁，不提供收入，也不会绕过金币解锁。经典地图继续保持当前地块显示尺寸；超出战斗视窗的部分使用鼠标拖动、单指拖动或滚轮查看。

当前开局生成规则：

| 区域/内容 | 规则 |
| --- | --- |
| 玩家大本营 | 位于本局随机出生模板对应的玩家侧出生点，开局已解锁 |
| 敌方大本营 | 使用同一出生模板的严格镜像点，开局已解锁 |
| 预归属 | 玩家与 AI 各 36 格，中线保留少量中立争夺格 |
| 真实解锁 | 除大本营外，所有地块都需要消耗金币解锁后才生成建筑/空地并计入拥有 |
| 地块站点 | 选择地图开局时按配置权重和本局种子重新生成问号、动物营地/大厅、防御塔或金矿；同一地图连续两局大部分普通地块应不同，对称地块的类型和价格必须一致 |
| 初始可解锁金矿 | 玩家大本营相邻可解锁地块中有且仅有 1 块金矿；敌方大本营相邻可解锁地块同样有且仅有 1 块金矿 |
| 地块价格 | 问号 25，金矿 50，营地按 50/100/250 价格池随机生成；防御塔按对应玩家本局购买次数使用 50、100、150……递增价；双方基地相邻各至少有 1 块 50 价格营地 |
| 营地卡牌 | 玩家和电脑解锁动物营地/大厅时，只能从各自当前出战编组中的动物卡选择；先匹配随机目标品质，没有时逐级向更低品质查找，绝不跨编组或向高品质补位 |
| 防御/金矿卡牌 | 解锁防御塔时只能从该玩家当前出战编组中的防御塔卡选择；先匹配随机目标品质，没有时逐级向更低品质查找，绝不跨编组或向高品质补位。目标品质及全部更低品质都没有时，该地块变为空地。金矿固定绑定紫色金矿卡 |

### 6.3 地块购买规则

玩家点击地块时按以下顺序处理：

1. 如果地块不可选中，则无操作。
2. 如果地块不属于己方视觉归属，不能购买，也不显示价格和隐藏建筑信息。

防御塔递增价格：

- 每个玩家或电脑独立记录本局主动购买的防御塔数量，开局计数为 0。
- 当前价格为 `50 + 已购买数量 * 50`，即第 1、2、3、4 座依次为 `50、100、150、200`。
- 只有购买明确标记为防御塔的地块并成功生成防御塔后才增加计数；问号地块随机翻出防御塔不增加计数。
- 战斗界面显示价、金币不足提示、玩家扣费和 AI 购买判断必须读取同一个动态价格，不能继续使用地图种子中的静态价格。
3. 如果地块未连接己方已解锁地块，提示未连接。
4. 如果金币不足，提示金币不足。
5. 扣除金币。
6. 根据地块类型生成建筑、空地或奖励反馈。

当前可购买条件：

- 目标地块必须有 `site`。
- 目标地块必须仍未真实解锁，不能已经属于任一阵营的真实拥有地块。
- 目标地块必须属于当前阵营的视觉归属；不可解锁地块不显示价格、站点图标或隐藏建筑信息。
- 目标地块不能已经有建筑。
- 目标地块必须连接己方已解锁地块或己方建筑。
- 玩家金币必须大于等于地块价格。

营地/防御塔显示和揭示规则：

- 解锁前只显示“动物营地”和价格，不提前显示具体是狼、袋鼠或其它动物营地，也不显示品质色。
- 解锁时先读取 `config/tables/cell_reveal_rules.csv` 中对应 reveal pool 的权重，决定本次结果类型；若结果为单位池或防御池，再读取 `config/tables/card_random_pools.csv` 中对应 card pool 的权重决定目标卡牌/品质。
- 营地地块按自身价格从 `config/tables/cell_price_pools.csv` 映射到 `unit_cards_price_50`、`unit_cards_price_100` 或 `unit_cards_price_250`，再按对应池子的概率抽取。
- 动物营地仅使用当前阵营出战编组内的动物卡：配置卡池只负责随机目标品质；目标品质有多张时按地块随机种子选择，没有同品质时按 `legendary -> epic -> rare -> common` 的方向逐级向下查找。不得跨编组、跨阵营或向更高品质补位；目标品质及全部更低品质都没有时，解锁结果为空地。
- 防御塔仅使用当前阵营出战编组内的塔卡，品质回退规则与动物完全一致：本品质优先，只向下取可用的最高品质，不得向上或跨编组；全部候选为空时生成空地。
- 解锁后的营地、防御塔和金矿会在地块上方短暂弹出对应卡牌；点击已解锁地块后，战斗区下方面板复用卡牌排版展示对应卡牌信息。
- 解锁后的营地可点击查看对应动物卡牌信息，防御塔显示塔卡攻击、生命、射程和冷却，金矿显示生命和金币收入；编组页详情使用简化属性图标展示关键属性。

### 6.4 地块占领规则

- 当前战斗规则不再允许动物把空地或普通地块作为攻击目标。
- 地块扩张主要通过玩家/AI 消耗金币解锁相邻站点完成。
- 动物完整离开一个敌方视觉归属地块时，如果该地块没有敌方动物且没有任何建筑，会把该地块软占领成经过方颜色；这只改变 `occupier`/`territory_team` 的战线显示，不清除站点、不生成建筑、不计入真实拥有地块，也不能替代金币解锁。
- 摧毁非基地建筑后，该建筑所在地块会转为攻击方真实拥有的空地，并播放反馈。
- 摧毁基地仍直接触发胜负，不进入普通地块占领流程。

### 6.5 建筑规则

| 建筑 | HP | 功能 |
| --- | ---: | --- |
| base | 50 | 基地，提供金币和固定低伤攻击，被摧毁则直接失败/胜利，不产兵 |
| barracks | 绑定动物最大生命 x 3 | 周期生成单位 |
| hall | 绑定动物最大生命 x 3 | 周期生成强单位 |
| tower | 读取绑定防御塔卡 | 每 1 秒攻击范围内最近的敌方单位或敌方建筑 |
| mine | 125 | 提供金币收入 |
| empty | 0 | 空地 |

当前攻击与生产：

| 建筑 | 周期/冷却 | 范围 | 伤害 | 说明 |
| --- | ---: | ---: | ---: | --- |
| tower | 1.0 秒（固定） | 绑定卡牌射程 + 0.5 格 | 读取绑定卡牌 | 半格按当前 `HEX_SIZE` 计算为 21.5 世界单位；卡牌升级不放大该固定增量 |
| base | 1.1 秒 | 210 | 2 | 双方基地统一攻击，不生产单位 |
| barracks / hall | 读取绑定动物召唤间隔 | - | - | 玩家和敌方使用相同生产间隔 |

战斗建筑视觉规则：

- 动物营地不再按动物物种做单独外形，解锁后只保留 common / rare / epic / legendary 4 种品质外形；未解锁时统一使用黑灰剪影。
- 防御塔同样只保留 4 种品质外形；未解锁时统一使用黑灰剪影。战斗属性读取绑定防御塔卡，射程统一额外增加半格，攻击间隔统一固定为 1 秒。
- 解锁后的建筑主色必须跟品质色一致：绿色、蓝色、紫色、金色。
- 每局按对局种子为玩家随机分配阵营色；颜色整体使用较低饱和度，避免大面积高饱和色造成视觉噪音。双边房间仍保持 `1-3` 为暖色组、`4-6` 为冷色组，自由混战使用六个等距色相。
- 同一玩家的预归属、软占领和真实金币解锁地块统一使用同一种低饱和阵营色，不再通过底色区分占领与解锁状态；当前可购买状态继续使用黄色描边表达。
- 六边形战斗地块整体按中间战斗框的中心自动定位，避免上方越界或下方空白过多。
- 营地、防御塔和金矿图标改为简化几何图标：动物营地是小房子，防御塔是细长塔，金矿是山形图标；未解锁时统一为中性色，不显示品质或具体卡牌颜色。
- 解锁后的动物营地和防御塔按对应卡牌品质上色；金矿按紫色金矿卡与金币强调色显示。
- 动物营地/大厅的召唤进度条使用更短、更细的小条，并放在六边形内部偏上位置；建筑生命条同样收窄。建筑生命值等于最大生命值时不显示生命条，首次受伤后才显示；多人槽位编号不受此规则影响并继续显示。战斗中的单位血条、建筑生命条和召唤进度条只保留一条底槽加填充，不额外绘制下方黑色阴影条。

### 6.7 当前单位原型

战斗内单位已经改为读取 `config/tables/cards.csv` 的动物卡配置：

| 单位 | HP | 速度 | 攻击 | 射程 | 冷却 | 搜敌 |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| 战斗单位 | 读取 `max_hp` | `move_speed x 50%` | 读取 `attack`，允许为 0 | 读取 `attack_range` | 1.70 | 敌对动物和敌方建筑 |

战斗动物按卡牌 `rarity` 叠加基础视觉倍率：

| 品质 | 品质色 | 动物视觉倍率 |
| --- | --- | ---: |
| common | 绿 | 100% |
| rare | 蓝 | 120% |
| epic | 紫 | 150% |
| legendary | 金 | 180% |

品质倍率只放大战斗中的动物本体和随身阴影，并以脚底为固定锚点；它会与攻击、受击、登场等程序化动作的瞬时缩放相乘。该倍率不改变单位世界坐标、生命、攻击、速度、射程、碰撞、寻路或寻敌，单位血条和多人队伍编号也保持固定界面尺寸。

注意：当前战斗单位为了放慢节奏，移动速度在战斗内统一乘以 0.5，攻击速度也降为原来的 50%。卡牌升级仍会按等级系数放大攻击、生命、速度、射程和召唤间隔。

### 6.8 单位技能落地

当前动物技能以 `config/tables/cards.csv` 为准：

- `skill_text` 有内容时，以文本规则优先落地；旧结构化字段不会覆盖文本技能。
- `skill_text` 为空时，使用 `skill_trigger / skill_effect / skill_power / skill_cooldown_sec` 作为兜底技能。
- 已接入触发：登场、攻击、受伤、阵亡、击杀、周期、参与占领、持续光环和目标偏好。
- 已接入效果：额外召唤、金币、攻击/生命成长、护盾、治疗/修复、减速、眩晕、反伤、斩杀、额外目标、范围伤害、速度光环、先手攻击、攻击/受击自损、附近友军承伤和建筑/后排/远程/前排优先攻击。
- 动物触发金币效果时，在该动物头顶显示金币图标与 `+数量`，短暂上浮并淡出；同一动物在极短时间内连续触发时合并数字，避免大量单位同时产金造成信息噪音。基地与金矿的周期基础收入不生成飘字。
- 技能数值是当前自动战斗原型的轻量化版本，例如 `skill_power` 会换算成小额护盾、治疗或额外伤害，避免旧大数值直接压垮当前低生命战斗节奏。
- 更复杂或带概率的 `skill_text` 会先按当前可识别关键词执行；后续如果要做到完全精确，应把这些文本技能拆成新的结构化字段。

### 6.9 单位 AI

单位每帧执行：

1. HP 小于等于 0 时移除并播放反馈。
2. 出生时选择最近的敌方建筑作为推进目标并缓存；只有该建筑死亡、消失或不再敌对时才重新选择。
3. 每次更新只扫描攻击范围内的敌对动物和建筑作为即时攻击候选，不使用范围外候选改变推进路径。
4. 如果范围内存在攻击目标，按冷却造成伤害；否则继续沿缓存路径向锁定建筑推进。
5. 所有动物移动速度按卡牌速度的 50% 执行，攻击冷却固定 1.70 秒。
6. 不追逐攻击范围外的动物，也不因途中出现更近建筑而重新寻路。
7. 远程攻击和防御塔攻击会生成一条短时间高亮弹道，帮助玩家看清攻击来源和目标。
8. 如果没有敌对动物或敌方建筑，则原地等待，不攻击空地。

### 6.10 敌方 AI

敌方 AI 当前规则：

- 开局 `4.0` 秒后首次尝试行动，之后每 `2.2` 秒尝试行动一次。
- 从可连接、可购买地块中选择目标。
- AI 与玩家一样使用战斗金币；如果敌方金币不足，则暂不购买。
- 购买后按同样地块翻开规则生成建筑。
- 敌方大本营不再产兵；敌方默认阵容为 8 张 common 绿色动物卡牌，产兵建筑按绑定卡牌召唤间隔生产。

### 6.11 胜负规则

| 情况 | 结果 |
| --- | --- |
| 玩家摧毁敌方基地 | 玩家胜利 |
| 敌方摧毁玩家基地 | 玩家失败 |
| 倒计时结束 | 按分数判定 |

倒计时分数：

```text
score = 拥有地块数 * 2 + 拥有建筑数 * 5
```

玩家分数大于或等于敌方分数时，玩家胜利。

结算奖励：

| 结果 | 奖励 |
| --- | ---: |
| 胜利 | 抽卡券 x3 |
| 失败 | 抽卡券 x1 |

### 6.12 段位匹配与镜像数据库

当前原型使用星数制段位。玩家资料保存当前段位 `rank_key` 和星数 `stars`，所有游戏页面只显示段位和星数，不显示原始隐藏分或数值变化。

经典单人 1V1 胜利固定提升 1 星，失败固定降低 1 星；多人自由混战和房间模式使用各自结算表中的星数变化。超过当前段位星数上限时晋级到下一段 1 星，低于 1 星时降到上一段满星。青铜 1 星失败不再降低；王者星数无上限，王者 1 星失败降到钻石 5 星。

| 段位 | 星级上限 | 晋级/降级规则 |
| --- | ---: | --- |
| 青铜 | 3 星 | 满星再胜进白银 1 星，1 星失败不再降低 |
| 白银 | 3 星 | 满星再胜进黄金 1 星，1 星失败降青铜 3 星 |
| 黄金 | 4 星 | 满星再胜进钻石 1 星，1 星失败降白银 3 星 |
| 钻石 | 5 星 | 满星再胜进王者 1 星，1 星失败降黄金 4 星 |
| 王者 | 无上限 | 胜利继续加星，1 星失败降钻石 5 星 |

匹配和数据规则：

- 玩家点击大厅“匹配”时，直接读取当前段位和星数，并优先匹配同段位镜像。
- 系统优先从 `user://rank_mirror_db.json` 中选择同段位其他玩家镜像；只有本机记录时退回本机镜像，没有同段位记录时生成同段位默认镜像兜底。
- 镜像记录包含段位、星数、8 张出战卡、卡牌等级和记录时间；旧 `elo` 字段只作为历史兼容元数据保留，不再驱动结算和界面显示。
- 经典单人 1V1 结算固定为胜利 +1 星、失败 -1 星；多人模式按名次或房间胜负使用独立星数奖励。
- 玩家在某段位胜利后，会把本局开战时的段位、星数、卡组和等级记录为该段位胜利镜像。
- 玩家失败只更新星数和胜负统计，不写入胜利镜像。
- 每个段位本地最多保留 20 条镜像记录，超过后移除最旧记录。

显示规则：大厅只显示段位、星数和胜负，不显示镜像数量等服务器/匹配池信息；经典单人战斗顶部显示双方段位星数，多人自由混战显示 3 分钟倒计时、玩家实时名次和六方地块榜，房间显示双方存活槽位数；结算页显示结算后的当前段位星数。

结算页玩家列表规则：结算时冻结本场全部参战玩家的最终顺序，每行显示名次、玩家名、结算前段位星级、星级增减和结算后段位星级。自己的结果卡固定在列表上方，以金色描边和“自己”标识特殊显示，进入结算页无需滚动即可看见；其余玩家放入独立纵向滚动区域，支持鼠标滚轮、鼠标拖动和触摸滑动。滚动区域不能带动战斗地图，也不能因拖动误触返回。

互联网房间由服务器在房间槽位快照中同步玩家进入房间时的 `rank_key` 与 `rank_stars`，各客户端根据最终名次或阵营结果计算并展示该玩家本场星级变化；AI 槽位使用系统初始段位。客户端不能用本机玩家段位冒充其他真人玩家的结算前段位。

当前版本使用本地 JSON 作为原型数据库。后续接入服务器时，可以把相同结构迁移为共享镜像库，匹配时从同段位其他玩家的胜利镜像中抽取对手。

### 6.13 大厅六人自由混战

- 点击大厅 `多人对战` 后直接创建 6 个独立阵营并进入战斗，不经过房间页；本地玩家固定为 1 号，其余 5 个槽位由电脑控制。
- 大厅 `多人对战` 与底栏 `房间` 是两个独立入口：前者只进入本地 6 人自由混战，后者才进入互联网双边房间，不得互相跳转替代。
- `多人对战` 按钮不直接展示 `6人乱斗 · 名次奖励` 等规则说明，仅在右上角显示小型 `HOT!` 角标，并与整个按钮共享同一点击区域。
- 自由混战使用专用 `ffa_regular_hex` 地图：地图是半径 10 格的标准等边正六边形，共 331 格；中心 1 格中立，其余 330 格按 60 度旋转对称分成 6 份，每名玩家预归属 55 格。该地图不进入 `3V3` 双边房间地图池。
- 每局从 3 套六重旋转对称的出生模板中随机 1 套，六个基地、起始金矿和起始营地由同一母版每 60 度旋转得到；所有玩家到中心和关键战线的相对距离一致。
- 地块类型和基础价格在选择该地图开局时按本局种子重新生成，再把母版结果六重旋转复制；因此同一张正六边形地图连续两局的大部分普通地块不同，但六方资源机会完全一致。
- 六个阵营各自为战，除同一阵营自身外不存在盟友；单位、基地和防御塔可以把任意其他存活阵营作为敌人。
- 自由混战独立使用 180 秒倒计时。战斗中按地块当前可见归属实时计分和排序，单位夺色会立即改变双方分数；同分时按槽位号从小到大排序，保证榜单稳定。
- 单个玩家的原始大本营被摧毁后，该玩家立即淘汰：该玩家单位清除，大本营修复为最后一击方拥有的大本营，其实际拥有或视觉控制的全部领地立即成为最后一击方的已解锁空地，除本次被攻破的大本营外的既有建筑全部清除；淘汰方地块积分归零，最后一击方实时获得这些地块积分。淘汰玩家不能继续购买地块，但本地玩家可留在战场观察实时排名。
- 倒计时结束时只按当前地块积分确定最终名次；同分时按槽位号从小到大排序。若倒计时前仅剩 1 名存活玩家，则提前按同一规则结算。
- 带有 `flying` 标签的可飞行动物可跨越地图空白区域；其他动物使用基于实际六边格邻接关系的简单寻路，移动路径中的每一步都必须落在地图已有地块上。
- 结算沿用 6 人名次奖励：第 1 名 `+3` 星与 5 张抽卡券并计 1 场胜利，第 2-3 名 `+2` 星与 2 张抽卡券，第 4-6 名 `-1` 星与 1 张抽卡券；第 2-6 名计入败场，结算后点击返回大厅。

### 6.14 双边房间模式

房间与地图规则：

- `1V1 / 2V2 / 3V3` 各有 5 张轮廓、通路和中央战线明显不同的地图，共 15 张；开局只在当前人数对应的 5 张地图中等概率随机。
- 所有地图按当前总人数严格等分，同局每一名玩家的领地形状、格数、基地相对位置、起始金矿和起始营地都能通过该地图的对称变换完全重合。`1V1` 为二等分镜像，`2V2` 为四等分对称，`3V3` 为六角形六等分并以 60 度旋转对应六个槽位。
- `1V1 / 2V2 / 3V3` 房间中的每名真人必须使用自己账号当前保存的 8 张出战卡组和对应卡牌等级。服务器在进入房间时从已认证账号资料读取 `deck` 与 `card_levels`，随槽位快照同步并在开战时冻结为该槽位的战斗卡组；房主客户端不能用房主卡组或统一敌方卡组覆盖其他真人。只有 AI 补位槽使用系统 AI 卡组。
- 每张地图提供 3 套合法出生模板；每局只随机 1 个模板编号，再对所有玩家做镜像、平移或旋转映射。不得为各玩家独立抽取不同模板。
- 选定地图后，按配置表规则和本局种子重新生成普通地块类型与基础价格；先生成一份母版，再对称复制到其他玩家区域。同地图连续开局应有过半普通地块的“类型+价格”组合发生变化。
- 每名玩家或电脑拥有 `30-100` 个初始预归属格，同图所有槽位数量一致；5 张同规模地图仍须有明显不同的外轮廓、通路或边界节奏。
- 每个基地保底 1 个相邻金矿和 1 个 50 金币营地；每名玩家的初始领地连通，整张地图也必须连通。
- 房间战斗和经典单人 1V1 共用轴向六边形世界坐标、邻接、命中检测和实际地图边界；屏幕点击先反算为世界坐标，绘制时再叠加摄像机偏移。

阵营与战斗规则：

- 我方使用槽位 `1, 2, 3`，对方使用槽位 `4, 5, 6`；按房间规模每边只启用前 N 个槽位。本地玩家固定为 1 号，未由真人接管的有效槽位由电脑控制。
- 槽位 `1-3` 使用每局随机的低饱和暖色组，槽位 `4-6` 使用每局随机的低饱和冷色组；同边颜色保持同一色温，通过色相与槽位编号区分具体玩家，不能把暖色和冷色混在同一边。每个槽位只生成一种阵营色，并统一用于预归属、软占领和真实解锁地块。
- 同一边互为盟友：单位、基地和防御塔不会攻击盟友，不会占领盟友地块，也不会把盟友基地作为推进目标。
- 每个槽位拥有独立金币、基地、单位、建筑、AI 计时器和存活状态；每个槽位最多同时保留 12 个战斗单位。
- 多人开局时为每个有效槽位建立独立的 8 张出战编组快照；默认电脑快照内容可以相同，但数组彼此独立。战斗中的动物与防御塔随机只能读取所属槽位快照，任何一个槽位的编组变化都不能污染其他槽位。
- 单个玩家的原始大本营被摧毁只淘汰对应槽位；最后一击方接管该大本营和该槽位全部领地，除本次被攻破的大本营外的败方建筑全部清除，避免攻击方获得编组外动物或防御塔。摧毁已经易主的副基地只让该基地再次易主，不淘汰当前拥有者，也不转移当前拥有者的其他领地。一边全部原始大本营被摧毁时战斗结束，另一边获胜。
- 房间局倒计时为 `300` 秒；超时后按双方存活基地数、基地总生命、拥有地块数、建筑数依次比较，完全相同则平局。

房间结算：

| 结果 | 星数变化 | 抽卡券 | 胜负统计 |
| --- | ---: | ---: | --- |
| 胜利 | +3 | 5 | 胜场 +1 |
| 平局 | 0 | 2 | 只增加场次 |
| 失败 | -1 | 1 | 败场 +1 |

星数变化允许跨段位连续结算；房间本地电脑局和后续真人房间都不写入单人 1v1 胜利镜像数据库。完整规则及验收口径见 `docs/ROOM_MODE_DESIGN.md`。

## 11. 推荐修改顺序

如果你要系统性改规则，建议按以下顺序进行：

1. 先改体验目标：决定一局时长、地图大小、玩家每局主要决策数量。
2. 再改经济节奏：初始金币、基地收入、金矿收入、地块价格。
3. 再改地块结构：问号、单位、防御、金矿比例。
4. 再改卡池品质：50/100/250 三档分别控制普通、稀有、史诗、传说概率。
5. 再改单位和防御塔：先保证每档都有清晰定位，再调数值。
6. 最后改 UI 呈现：根据最终规则决定哪些信息必须在大厅、编组和战斗中显示。

## 13. 每次修改后的验证

修改配置表后：

```powershell
python tools/validate_config.py
python tools/export_config.py
```

`config/tables/*.csv` 是设计源表，Godot 实际运行读取 `runtime/config/*.json`。因此任意 CSV，尤其是 `config/tables/cards.csv`，修改后必须立刻校验并导出 runtime JSON，否则游戏内仍会显示上一版导出的数值。

如果系统没有 `python` 命令，可以使用：

```powershell
py -3 tools/validate_config.py
py -3 tools/export_config.py
```

修改 Godot 脚本后：

- 打开 Godot 项目确认主场景可运行。
- 检查大厅、编组、战斗、结算流程。
- 至少测试一次胜利、一次失败、一次倒计时结算。
- 编组页测试卡牌替换、页签切换、滚动和锁定按钮提示。
