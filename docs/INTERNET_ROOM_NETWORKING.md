# 公网房间联网规格

> 状态：首版实现基线。目标是让不同网络中的玩家通过公网加入玩家创建的 `1V1 / 2V2 / 3V3` 房间。本文只定义联网边界和验收口径，不改变现有房间地图、联盟、战斗、结算与淘汰规则。

当前已落地：ENet 客户端/专用服务器、权威房间码与槽位、创建/加入/换位/准备/开局、房主命令路由、可靠完整战斗快照、房主离开后的权威迁移，以及主流程真人槽位操作。当前验证范围为本机真实 ENet 回环；只有把服务端部署到公网并完成双网络实机验收后，才可标记“公网已验证”。

## 1. 目标与非目标

### 1.1 本期目标

- 房主创建双边房间并取得短房间码。
- 其他正在游玩的玩家连接同一公网服务，输入房间码后加入可用真人槽位。
- 支持 `1V1`、`2V2`、`3V3`，最多 6 名真人；房主可按现有规则启用电脑补位。
- 所有客户端只向公网服务器建立出站 UDP 连接，玩家不需要公网 IP、端口映射或互相暴露地址。
- 公网服务器负责房间注册、槽位分配、准备状态、开局校验和战斗消息路由。
- 首版由房主客户端执行战斗模拟，其他客户端提交命令并接收经服务器转发的权威快照。

### 1.2 本期非目标

- 不做公开房间列表、自动匹配、好友系统、账号体系、观战和中途补位。
- 不做跨版本兼容。当前代码尚未实现协议版本、游戏版本或配置哈希握手，因此客户端与服务器必须由部署流程保证使用同一构建；自动拒绝不一致版本属于上线前增强。
- 不将完整战斗模拟放到服务器，也不承诺竞技级防作弊和掉线后的无损房主迁移。
- 不修改现有房间模式的地图池、槽位编号、联盟、胜负、奖励和电脑逻辑。

## 2. 选定架构

### 2.1 拓扑

```text
加入者 A ─┐
加入者 B ─┼─ ENet/UDP 24567 ─ 公网 Godot headless 服务器 ─ ENet/UDP 24567 ─ 房主客户端
加入者 C ─┘                         │
                                     └─ 房间注册、槽位、准备、开局、路由
```

- 公网节点运行 Godot 4 headless，监听 `0.0.0.0:24567/UDP`。
- 客户端使用 `ENetMultiplayerPeer` 连接公网节点，不建立玩家到玩家的直连。
- 玩家创建房间后成为“逻辑房主”，拥有切换房间规模、电脑补位和发起开局的权限。
- 公网服务器才是网络主机：它生成房间码和 `match_id`、维护成员与槽位，并拒绝越权消息。
- 战斗开始后，逻辑房主客户端同时是首版“战斗模拟权威”。服务器将其他玩家命令路由给房主，再将房主快照转发给房间内所有客户端。

### 2.2 原型公平性限制

房主客户端掌握首版战斗状态、随机推进和合法性判定，因此被修改的房主程序可以伪造金币、单位、伤害或结算。当前公网服务器会验证 ENet 发送者、房间成员、槽位、房主权限、命令/快照大小，以及权威快照发送者和递增序号；命令动作和战斗合法性仍由房主客户端判断。当前代码没有限频、账号鉴权或服务器侧战斗复算，无法证明每个战斗结果都符合游戏规则。

因此首版只用于受信玩家测试或非严肃竞技：

- 不把客户端权威结果作为高价值排行、付费资产或不可撤销竞技奖励的唯一依据。
- 房主权威断线时会把战斗权威迁移给剩余真人，由新权威从自己最后收到的完整快照继续；这是尽力恢复，可能丢失最后一份已接收快照之后的进度，不应据此发放高价值竞技奖励。
- 后续正式竞技版本必须把固定步长战斗模拟、随机数和结果判定迁移到专用服务器；客户端只提交输入并显示服务器状态。

## 3. 网络模块边界

建议新增独立模块，避免把传输细节继续写入 `scripts/app/main.gd`：

| 模块 | 建议位置 | 职责 | 不负责 |
|---|---|---|---|
| 服务端入口 | `scenes/server.tscn`、`scripts/server/server_main.gd` | 启动 ENet、加载配置、维护连接 | 战斗模拟与 UI |
| 房间注册表 | `scripts/network/room_registry.gd` | 创建/加入/离开、槽位、准备、房主迁移、空房关闭 | 地图与战斗规则 |
| 消息协议 | `scripts/network/room_protocol.gd` | 槽位、房间码和输入字段校验 | 业务状态存储 |
| 客户端/服务端路由 | `scripts/network/online_room.gd` | ENet、RPC、发送者鉴权、命令和快照转发 | 信任客户端给出的身份 |
| 房间 UI 与战斗桥接 | `scripts/app/main.gd` | 网络事件映射到房间槽位、本地命令和权威快照 | 保存房间网络真相 |

房间注册表是大厅真相源；战斗开始后，服务器继续掌握成员和槽位，房主客户端只掌握战斗状态。客户端不得通过直接修改 `room_human_teams` 冒充已联网玩家。

## 4. 房间与槽位规则

### 4.1 固定槽位

沿用现有编号，不在联网层重新排序：

| 模式 | 我方 A | 对方 B | 最大真人数 |
|---|---|---|---:|
| `1V1` | `1` | `4` | 2 |
| `2V2` | `1, 2` | `4, 5` | 4 |
| `3V3` | `1, 2, 3` | `4, 5, 6` | 6 |

- 房主默认占用 A 方 1 号槽位。
- 加入者可申请有效空槽；若未指定，服务器按 A 方空槽、B 方空槽的稳定顺序分配。
- 槽位分配由服务器原子执行；两人同时申请同一槽位时只有一个成功。
- 已占用、已锁定、无效或超出当前规模的槽位不可加入。
- 房主缩小模式前，目标规模外不得存在真人；否则服务器拒绝变更并返回占用槽位。
- 开局后锁定模式、队伍、槽位、地图和电脑补位，不允许换边或新玩家加入。

### 4.2 房间身份

- `room_code`：服务器生成且在当前进程内唯一的 6 位数字码；只用于定位房间，不等于安全认证令牌。
- `peer_id`：当前 ENet 连接 ID，只能由服务器绑定，客户端不能自报或覆盖。
- `team_id`（槽位 ID）：`1..6`，由服务器分配并随房间状态广播。

房间码在最后一名真人离开或服务器重启后失效。首版没有账号身份和重连令牌。

### 4.3 准备与开局

- 每个真人槽位都有 `joined / ready` 状态；房主不能替其他真人设为准备。
- 房主创建后默认未准备，所有真人必须主动准备。
- 若电脑补位关闭，所有有效槽位必须由已准备真人占用。
- 若电脑补位开启，至少要有房主，真人槽位全部准备后，剩余有效槽位由房主模拟的电脑占用。
- 只有逻辑房主可以调用 `_rpc_request_start_room`。
- 服务器再次检查房间状态，生成本进程内本局使用的 `match_id` 和确认后的地图 ID，再通过 `_rpc_receive_match_started` 通知所有真人。
- 首版收到开局通知后立即加载同一地图；客户端加载确认与超时回滚留作后续增强。

## 5. 房间生命周期

```text
连接服务器 -> 创建/加入房间 -> 分配槽位 -> 准备 -> 锁定房间
      -> 下发 match_id / map_id -> 开始战斗
      -> 命令与快照转发 -> 玩家离开 -> 房间关闭或房主迁移
```

首版服务端使用两个持久状态：`lobby` 允许加入、离开、换槽、准备和房主设置；`running` 锁定加入与设置，只接受本局命令、权威快照、离开和权威迁移。最后一名真人离开时释放房间码。

## 6. 消息协议

### 6.1 RPC 边界

首版通过固定白名单 RPC 传输 Godot 基础类型、数组和字典，不接收对象、资源路径或任意方法名。`sender_peer_id` 不接受客户端字段；服务器始终从 `multiplayer.get_remote_sender_id()` 注入身份。房间写操作统一返回 `ok/error`，房间变化统一广播完整快照。

### 6.2 控制 RPC（可靠有序，通道 0）

下表使用当前脚本中的实际 RPC 方法名；公开协议若后续改名，应同步更新代码、测试和本文。

| RPC | 方向 | 关键参数/字段 | 当前行为 |
|---|---|---|---|
| `_rpc_request_create_room` | 客户端→服务器 | `player_name, players_per_side, options` | 创建房间；调用者占 A1；`options` 只保留规模、AI 补位和有效地图 ID |
| `_rpc_request_join_room` | 客户端→服务器 | `room_code, player_name` | 按固定优先级分配空槽；AI 槽可被真人替换 |
| `_rpc_request_update_room_options` | 客户端→服务器 | `options` | 仅房主可修改规模、AI 补位和地图选项 |
| `_rpc_request_move_to_slot` | 客户端→服务器 | `team_id` | 只移动发送者到当前模式的空槽或 AI 槽 |
| `_rpc_request_set_ready` | 客户端→服务器 | `ready` | 只修改发送者的准备状态 |
| `_rpc_request_leave_room` | 客户端→服务器 | 无 | 释放发送者槽位；必要时迁移房主/战斗权威或关闭空房 |
| `_rpc_request_start_room` | 客户端→服务器 | 无 | 仅房主可调用；检查有效槽位和真人准备状态 |
| `_rpc_receive_operation_result` | 服务器→客户端 | `operation, result{ok,error}` | 返回当前实现使用的成功结果或错误字符串 |
| `_rpc_receive_room_snapshot` | 服务器→客户端 | `room_code, status, players_per_side, fill_with_ai, host_peer_id, local_team_id, revision, slots, options, map_id` | 房间变化后向房间真人广播完整快照 |
| `_rpc_receive_match_started` | 服务器→客户端 | `match_id, map_id, local_team_id, authority_peer_id, is_authority, room_snapshot` | 锁房并让所有真人进入服务器选定的同一地图 |
| `_rpc_receive_authority_changed` | 服务器→客户端 | `match_id, authority_peer_id, is_authority` | 原权威离开后通知剩余真人接管关系 |

### 6.3 战斗命令（可靠有序，通道 2）

非房主客户端把玩家意图发送给服务器：

```gdscript
{
    "action": "unlock_tile",
    "sequence": 103,
    "q": 4,
    "r": -2
}
```

- 服务器从发送者当前所在房间取得本局 `match_id`，并把连接绑定的 `sender_peer_id / sender_team_id` 附加到命令信封后转发给当前权威；客户端不能自报这三个身份字段。
- 当前只接入 `unlock_tile`。客户端命令不包含阵营、金币、伤害、结果或其他权威值。
- 房主按 `sender_peer_id + sequence` 丢弃重复/旧命令，再按 `sender_team_id` 和当前战斗状态验证解锁。
- 房主自己的操作也走同一命令入口，避免本地路径与远端路径产生两套规则。
- 首版不做回滚；客户端只可立即显示选择框、按压等非权威反馈，资源与战场状态以房主快照为准。

### 6.4 战斗快照（首版可靠有序，通道 1）

- 房主以固定 `20 tick/s` 推进模拟，首版以 `5 snapshot/s` 发送完整状态。
- 完整快照大于常见 UDP MTU，因此使用 ENet 可靠分片；不能把当前完整字典直接改成非可靠单包。
- 快照包含序号、房间计时、队伍资源、全部地块、建筑/单位状态、淘汰和结算状态。
- 服务器只接受当前房间的当前权威发送者和递增序号，并用服务器保存的 `match_id` 封装后转发给同房间真人；客户端还会检查信封及快照内的 `match_id`。
- 加入者丢弃旧快照，并且不在本地推进权威战斗。
- 后续带宽优化可拆分为低频可靠完整检查点和高频非可靠增量；首版依赖 ENet 可靠传输，在连接保持有效且重传完成时收敛到最新完整状态。

## 7. 断线与权威迁移

- ENet 报告连接离开或断开后，服务端立即从房间注册表移除该 Peer；首版不保留重连宽限期。
- 大厅阶段房主离开时，房主权迁移给加入时间最早的剩余真人；无人时立即关闭房间并释放房间码。
- 战斗阶段房主离开时，服务器把权威迁移给剩余真人；新权威以自己最后收到的完整快照继续固定步长模拟，不重新开局。
- 非房主离开后，其槽位不接受新玩家中途加入；当前房主继续推进战斗。
- 公网服务器重启或不可达时，所有客户端清除本局并返回联网房间入口；首版不承诺跨服务器进程恢复。
- 会话令牌、宽限期原槽位恢复和服务端战局持久化属于后续增强，不能在首版 UI 中宣称已经支持。

## 8. 配置与部署

### 8.1 服务端配置

| 配置 | 默认值 | 说明 |
|---|---:|---|
| `bind_host` | `0.0.0.0` | 服务端监听全部公网/内网接口 |
| `server_host` | `127.0.0.1` | 客户端默认连接地址，生产环境必须覆盖 |
| `server_port` | `24567` | ENet UDP 端口 |
| `max_clients` | `64` | 单进程连接上限，可在 `2..512` 调整 |
| `max_command_bytes` | `65536` | 战斗命令上限 |
| `max_snapshot_bytes` | `1048576` | 权威快照上限 |

服务端和客户端从 Godot 命令行参数或环境变量读取配置。客户端支持 `--server-host/--server-port` 或 `ZHANCHENG_SERVER_HOST / ZHANCHENG_SERVER_PORT`；服务端支持 `--bind-host/--server-port/--max-clients` 或 `ZHANCHENG_BIND_HOST / ZHANCHENG_SERVER_PORT / ZHANCHENG_MAX_CLIENTS`。

### 8.2 启动命令

开发环境从项目目录启动专用服务端场景：

```powershell
& 'D:\Program Files (x86)\Godot_v4.6.2-stable_win64_console.exe' `
  --headless --path . --scene res://scenes/server.tscn `
  --log-file tmp/network_server.log -- `
  --bind-host=0.0.0.0 --server-port=24567 --max-clients=64
```

当前仓库提供 `Windows Dedicated Server` 导出预设；资源包生成已验证，完整 EXE 仍要求本机安装同版本 Windows 导出模板：

```powershell
& 'D:\Program Files (x86)\Godot_v4.6.2-stable_win64_console.exe' `
  --headless --path . `
  --export-release 'Windows Dedicated Server' `
  build/windows/JungleLawServer.exe

& '.\build\windows\JungleLawServer.exe' --headless -- `
  --bind-host=0.0.0.0 --server-port=24567 --max-clients=64
```

专用导出通过 `dedicated_server` feature 让 `OnlineRoom` 自动启动监听，不依赖客户端主场景。若生产主机使用 Linux，需要先安装同版本 Linux/BSD 导出模板并增加对应预设；联网协议和启动参数保持不变。

客户端开发启动示例：

```powershell
& 'D:\Program Files (x86)\Godot_v4.6.2-stable_win64_console.exe' `
  --path . -- --server-host=game.example.com --server-port=24567
```

部署要求（运维侧待落实；当前仓库没有实现业务事件日志、进程守护或日志轮转）：

- 云主机安全组和系统防火墙放行入站 `UDP 24567`；开放同号 TCP 端口不能替代 UDP。
- 域名 A/AAAA 记录指向公网服务端；客户端默认连接域名，不把 IP 硬编码到战斗脚本。
- 服务进程以非管理员用户运行，工作目录只开放必要的日志写权限。
- 增加最小化业务事件日志，可记录连接 ID、房间码/比赛 ID 和错误类别；不要记录完整 IP 展示、原始网络包或客户端隐私数据。当前实现没有会话令牌和聊天系统。
- 进程由 systemd、容器编排或等价守护程序拉起；异常退出要能自动重启并保留轮转日志。

## 9. 安全边界

### 9.1 当前已实现

- 所有客户端请求都由服务端读取 `multiplayer.get_remote_sender_id()`；房间注册表再把该连接绑定到房间和 `team_id`。战斗命令中的发送者与阵营由服务端注入。
- 房间注册表校验 `peer_id`、玩家名、6 位数字房间码、`players_per_side`、布尔选项和 `team_id`，并按房间状态及房主权限拒绝越权操作。
- `online_room.gd` 只暴露固定 RPC 方法；房间设置只保留规模、AI 补位和有效地图 ID，不接受任意方法名或资源路径。
- 战斗命令和完整快照分别限制为 `65536` 与 `1048576` 字节；快照只接受当前权威发送者和递增序号。
- 房主按每个发送 Peer 的命令 `sequence` 丢弃重复/旧命令；访客按快照 `sequence` 丢弃重复/旧快照。
- 房间码在最后一名真人离开或服务端进程重启后失效；开局后房间注册表拒绝加入、换槽、准备和设置变更。

### 9.2 当前未实现，上线前必须补强

- 协议版本、游戏版本和配置哈希握手；当前不同构建之间没有自动拒绝机制。
- 连接、建房、尝试房间码、准备、命令和快照的限频/封禁；当前没有 `RATE_LIMITED` 响应或源地址冷却。
- `request_id / command_id` 去重、请求确认窗口和断线重连；当前只有命令/快照整数序号去重。
- 账号身份、会话令牌、重连令牌、原槽位宽限期与跨服务端进程恢复。
- 密码学安全的邀请凭据。当前 6 位房间码由 `RandomNumberGenerator` 生成，只是定位码，不是密码。
- 传输加密、DDoS/流量清洗、隐私审计、服务端战斗复算和竞技级反作弊。

### 9.3 已知限制

- 原生 ENet 连接不等同于端到端加密和账号认证，房间码也不是密码。
- 房主模拟权威可以作弊，服务端转发快照并不代表服务端验证了战斗合法性。
- 权威迁移仅从新权威最后收到的可靠完整快照继续，不能保证零丢帧、零回退或公平结算。
- 若需要正式运营，应在公网开放前完成 9.2，并把高价值战斗迁移到专用服务端权威。

## 10. 当前错误字符串

当前协议没有统一的大写错误码枚举。`operation_failed` 直接传递房间注册表或传输层返回的字符串；`main.gd` 对常见房间错误显示中文提示，其余错误直接显示原字符串。

| 当前错误字符串 | 触发条件 | 当前客户端处理 |
|---|---|---|
| `room_not_found` | 6 位房间码不存在或已失效 | 提示房间码不存在/失效，留在联网入口 |
| `room_full` | 当前模式没有空槽或可替换 AI 槽 | 提示房间已满 |
| `room_running` | 已开局后尝试加入或修改大厅状态 | 提示房间已经开战 |
| `host_only` | 非房主修改设置或开局 | 提示只有房主可操作 |
| `players_not_ready` | 有真人未准备 | 提示仍有真人未准备 |
| `room_not_full` | AI 补位关闭且有效槽未填满 | 提示真人未满且未启用电脑补位 |
| `room_size_conflict` | 缩小规模会排除现有真人 | 提示目标规模外仍有真人 |
| `slot_occupied` | 目标槽位已有真人 | 提示槽位已被占用 |
| `inactive_team_slot` | 目标槽位不属于当前模式 | 提示槽位未启用 |
| `peer_already_in_room / peer_not_in_room` | 重复加入房间或未加入就操作 | 当前没有专用中文映射，显示原字符串 |
| `invalid_* / *_too_long` | 类型、范围、房间码或名称校验失败 | 当前没有专用中文映射，显示原字符串 |
| 中文传输错误文本 | 未连接、命令/快照过大、比赛未开始、非权威发送快照等 | 直接显示文本 |

`VERSION_MISMATCH`、`CONFIG_MISMATCH`、`RATE_LIMITED`、`SESSION_EXPIRED` 和 `HOST_AUTHORITY_LOST` 当前均未实现，也不会由服务端发出。

## 11. 验收测试

### 11.1 当前自动化覆盖

- 协议与注册表：字段类型/范围、玩家名、6 位数字房间码、房间码样本唯一性、快照深复制和空房关闭。
- 槽位矩阵：`1V1 / 2V2 / 3V3` 分别只开放 `1,4`、`1,2,4,5`、`1..6`，并覆盖 AI 补位与真人替换 AI。
- 权限与状态：非房主不能改设置或开局，缩小规模冲突为原子失败，准备门禁与开局后锁定有效。
- 真实 ENet 回环：1V1 创建、凭码加入、换位、准备、开局、统一地图、可靠命令、可靠完整快照，以及伪造非权威快照被拒绝。
- 主流程桥接：访客控制服务器分配的 4 号槽位、地块解锁、计时/资源/结算快照、访客不自行推进战斗，以及房主离开后的权威迁移。
- 容量保护：生成的 3V3 完整快照低于当前 `1048576` 字节传输上限。

当前自动化没有覆盖：版本/配置握手、限频、源地址冷却、真正并发的同槽竞争、会话重连、网络故障注入、六名真人 3V3、公网 NAT 路径和长时间压力。

### 11.2 本机集成

- 真实 ENet 回环覆盖 1V1 双客户端创建、凭码加入、准备、开局、统一地图、命令转发和可靠权威快照。
- 2V2、3V3 覆盖 AI 补位创建、准备、开局、统一地图和退出。
- 主流程回环覆盖访客控制服务器分配的 4 号槽位、地块解锁同步、计时同步和房主离开后的权威迁移。
- 公网延迟、丢包、六台独立设备和长时间压力仍必须在部署后补做，不能以本机回环替代。

### 11.3 公网验收

- 服务端部署在具有公网地址的主机，仅开放 `UDP 24567`。
- 至少两名测试者位于不同运营商或一人使用家庭宽带、一人使用移动网络；双方无需端口映射即可连接。
- 房主创建房间后，远端玩家通过房间码加入并看到同一槽位状态；建议将正常网络下端到端加入耗时目标设为 `10s`，实测后再固化为发布门槛。
- 用 6 台真实客户端完成一局 `3V3`；战斗期间无跨房间串包、重复身份或槽位错位。
- 记录 30 分钟压力数据：连接数、房间数、收发带宽、平均/95 分位延迟、丢包、服务端 CPU/内存和错误率。
- 服务端重启后旧房间码全部失效，客户端能明确返回联网入口，不出现假在线。

## 12. 完成定义与后续迁移

代码完成定义：

- 专用服务端可按上述命令启动并监听 `UDP 24567`。
- 房间创建、码加入、固定槽位、准备、开局和权威迁移均由服务器管理。
- 房主命令入口、远端命令入口和既有本地战斗规则复用同一执行路径。
- 房间注册、真实 ENet 传输和主流程战斗回环自动化通过。
- UI 明确显示连接、创建、输入房间码、加入、槽位、准备、离开与开局状态。

公网发布完成定义还必须额外满足：服务端部署到固定公网地址，云安全组与系统防火墙放行 UDP 24567，并由不同网络上的真实客户端完成 1V1、2V2、3V3 验收。当前仓库未绑定具体公网主机，因此代码完成不等于公网部署已经完成。

正式竞技迁移时可保持房间和槽位模型，把战斗命令的目标从房主客户端改为专用战斗实例，把战斗快照与结算来源改为该实例。这样可复用房间 UI、房间码、准备和开局流程，同时移除首版最主要的公平性风险；版本握手和重连流程需另行设计与实现。
